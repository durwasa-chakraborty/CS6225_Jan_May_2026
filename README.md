# CS6868 :: Programs and Proofs

## Structure 

``` 
.
├── _CoqProject          - logical namespace mappings + file list
├── Makefile             - generated by coq_makefile, do not edit
├── Makefile.conf        - generated by coq_makefile, do not edit
├── plf/                 - PLF namespace: Maps.v, Imp.v
├── lectures/            - Lectures namespace: W1–W10, W9a, W9b
├── labs/                - Labs namespace: lab .v files go here
└── assignments/         - Assignments namespace: assignment .v files go here
```

### Namespace mapping (`_CoqProject`)

| Directory      | Logical Namespace |
|----------------|-------------------|
| `plf/`         | `PLF`             |
| `lectures/`    | `Lectures`        |
| `labs/`        | `Labs`            |
| `assignments/` | `Assignments`     |

`plf/` and `lectures/` are **separate namespaces**. Imports must reflect this:

```coq
From PLF      Require Import Maps.   (* plf/Maps.v *)
From PLF      Require Import Imp.    (* plf/Imp.v  *)
From Lectures Require Import W7.     (* lectures/W7.v *)
From Lectures Require Import W9a.    (* lectures/W9a.v *)
```

## Adding a new lab or assignment

1. Create `labs/LabN.v` (or `assignments/HwN.v`)
2. Append the path to `_CoqProject`
3. Regenerate and build (see below)

### Lab file boilerplate

```coq
Set Warnings "-notation-overridden".
From PLF      Require Import Maps.
From PLF      Require Import Imp.
From Lectures Require Export W7.
Require Import Bool.
Require Import Arith.
Require Import EqNat.
Require Import PeanoNat. Import Nat.
Require Import Lia.
```

## Building

### First time / after editing `_CoqProject`

```bash
coq_makefile -f _CoqProject -o Makefile && make
```

`coq_makefile` generates two files:
- **`Makefile`** — the actual build rules
- **`Makefile.conf`** — coq path configuration used by the Makefile

Both are generated artifacts. Do not edit them by hand; re-run `coq_makefile` whenever `_CoqProject` changes.

### Clean and rebuild everything

```bash
make clean && make
```

### Check that `coq_makefile` is available

```bash
which coq_makefile
```

If this returns nothing, `coq_makefile` is not on your `PATH`. Fix by sourcing the Coq Platform environment or checking your opam switch.

## Pitfalls

### 1. Coq version mismatch (most common error)

`.vo` files are compiled against a specific version of `coqc`. If your local `coqc` version differs from what compiled a `.vo`, you will see:

```
Error: compiled library Foo makes inconsistent assumptions over library Bar
```

or

```
The file foo.vo was compiled with version X but this is version Y
```

**Fix:** ensure everyone on the project uses the same Coq Platform release. Check yours with:

```bash
coqc --version
```

Agree on a version and pin it in your opam switch. Do not mix `.vo` files compiled by different `coqc` versions.

### 2. Importing `W*` files with the wrong namespace

All `W*.v` files live in `lectures/` under the `Lectures` namespace. Using `From PLF Require Import W7` will fail because `W7.v` is no longer in `plf/`.

```coq
(* WRONG *)
From PLF Require Import W7.

(* CORRECT *)
From Lectures Require Import W7.
```

### 3. Editing `_CoqProject` without re-running `coq_makefile`

Adding a file to `_CoqProject` does not update the `Makefile` automatically. Always re-run:

```bash
coq_makefile -f _CoqProject -o Makefile
```

### 4. Build order

`coq_makefile` infers build order from `Require` dependencies. `plf/Maps.v` and `plf/Imp.v` compile first, then `lectures/`, then `labs/` and `assignments/`. If a file is missing from `_CoqProject`, it will not be compiled and dependents will fail.

### 5. Finding all `Require` references to `W*` files

If you are auditing imports after a restructure:

```bash
grep -rn "Require.*W" --include="*.v" .
```

This shows every file and line that imports a `W*.v` module so you can verify all namespaces are correct.
